## Shadowing passage

> One of the things I take most seriously as a senior engineer is growing the people around me. Let me share how I approach mentoring. First, I set a growth goal in the first week, not after a performance review. I sit down and ask: what do you want to be able to do in six months that you can't do now? I write it down. Everything I assign from that point is calibrated to move them toward that goal. Second, I use code review as a teaching tool, not a gating tool. For juniors, I always separate blocking feedback from improvement feedback. I make it clear which comments must be addressed before merge and which are suggestions for their learning. I also explain the 'why' — not 'you should use a cache here' but 'you should use a cache here because without it, this endpoint will hit the database on every request and won't survive the traffic load we're building toward.' Third, I give feedback in the smallest useful increment. Not 'you need to think more about edge cases' but 'what happens to this function if the input array is empty?' Specific and immediate. Fourth, I pair-program on hard problems. The goal isn't to show them the solution — it's to show them how I think: how I break a problem down, when I look at documentation, when I admit I don't know and Google it. Process modeling is more valuable than any individual answer.
