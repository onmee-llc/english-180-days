## Topic 1 — Introduce Yourself as a Senior Engineer
**Weeks 1–2 | Daily practice: 30 min**

### Why This Topic First
This is the most-used response in every interview. It must feel completely natural — not memorized, but internalized. Anh sẽ dùng nó ít nhất một lần trong mỗi cuộc phỏng vấn.

### Content Outline
**Structure of a strong self-introduction (90–120 seconds):**
1. Name + years of experience + core role
2. Technical specialization (backend + AI/automation)
3. Most relevant recent work (1–2 sentences)
4. What you are looking for now
5. Why you are excited about this opportunity

**Sample Script (adapt to your own words):**
> "I'm Robert, a backend and AI engineer with over 12 years of experience. My core strength is building scalable backend systems and integrating AI/LLM capabilities into real-world products — from automation workflows to content generation pipelines. Most recently, I've been building a cloud-based SaaS tool that auto-generates niche content and schedules Pinterest pins, using Claude API and image generation models. I'm now looking for a senior remote role where I can bring both deep backend experience and practical AI integration skills to a strong engineering team."

### Key Vocabulary
- Senior backend engineer / AI integration specialist
- Scalable systems / distributed architecture
- LLM integration / prompt engineering / AI pipelines
- Automation workflows / content generation
- Remote-first team / async communication
- 12+ years of experience / hands-on background
- Looking for / seeking / open to

### Practice Questions
1. Tell me about yourself.
2. Walk me through your background.
3. What kind of role are you looking for?
4. Why are you leaving your current job?
5. What makes you different from other senior engineers?

### Week 1 Goal
Record yourself answering "Tell me about yourself" — aim for 90 seconds, natural pace, no reading.

### Week 2 Goal
Answer all 5 practice questions without preparation. Each answer should feel automatic.

## Topic 2 — System Design Fundamentals
**Weeks 3–4 | Daily practice: 30 min**

### Why This Topic
System design interviews are standard for senior roles at companies paying $5,000+/month. Anh cần nói được tư duy kiến trúc bằng tiếng Anh, không chỉ vẽ diagram.

### Content Outline
**Core concepts to be able to explain verbally:**
1. **Scalability** — horizontal vs vertical scaling, when to use each
2. **Load balancing** — round robin, least connections, sticky sessions
3. **Database design** — SQL vs NoSQL, when to choose which, indexing
4. **Caching** — Redis, CDN, cache invalidation strategies
5. **Message queues** — async processing, Kafka, RabbitMQ use cases
6. **CAP Theorem** — consistency, availability, partition tolerance trade-offs
7. **API Gateway** — rate limiting, authentication, routing
8. **Monolith vs Microservices** — when to split, when not to

**Framework for answering system design questions:**
1. Clarify requirements (functional + non-functional)
2. Estimate scale (users, requests per second, storage)
3. Design high-level architecture
4. Deep dive into critical components
5. Discuss trade-offs and alternatives

### Key Vocabulary
- Bottleneck / throughput / latency / availability
- Single point of failure / fault tolerance / redundancy
- Read-heavy / write-heavy workload
- Eventual consistency / strong consistency
- Horizontal scaling / sharding / partitioning
- Trade-off / constraint / assumption
- "Let me clarify the requirements first..."
- "The bottleneck here would be..."
- "I'd choose X over Y because..."

### Practice Questions
1. Design a URL shortener (like bit.ly)
2. Design a content scheduling system (relevant to your Pinterest SaaS)
3. How would you scale a system from 1,000 to 1,000,000 users?
4. What is the CAP theorem and when does it matter in practice?
5. When would you choose MongoDB over PostgreSQL?

### Week 3 Goal
Present the URL shortener design out loud in 5 minutes — whiteboard-style thinking in English.

### Week 4 Goal
Answer any of the 5 practice questions with a structured 3–5 minute verbal explanation.

### Content Outline

- Service mesh / sidecar proxy
- Circuit breaker / retry logic / backoff strategy
- Contract-first design / API versioning
- "This endpoint handles..." / "The response returns..."

### Practice Questions

2. How do you handle authentication in a microservices architecture?

## Topic 5 — AI/ML Pipelines & LLM Integration
**Weeks 9–10 | Daily practice: 30 min**

### Why This Topic Is Your Biggest Advantage
Most senior backend engineers cannot speak fluently about LLM integration, prompt engineering, and production AI pipelines. This is where anh differentiate mạnh nhất — nhưng phải trình bày được rõ ràng bằng tiếng Anh.

### Content Outline
1. **LLM integration patterns** — API calls, streaming responses, function calling, tool use
2. **Prompt engineering** — system prompts, few-shot examples, chain-of-thought, output formatting
3. **AI agents** — what they are, how they work, when to use them
4. **RAG (Retrieval-Augmented Generation)** — vector databases, embedding models, retrieval strategies
5. **AI pipeline design** — input processing, model calls, output validation, error handling
6. **Evaluation and testing** — how to measure AI output quality, A/B testing prompts
7. **Production considerations** — latency, cost per call, fallback strategies, rate limits

### Key Vocabulary
- Large Language Model / foundation model / fine-tuning
- Prompt / system prompt / user prompt / completion
- Token / context window / temperature / top-p
- Embedding / vector store / semantic search
- Agentic workflow / tool calling / function calling
- Hallucination / grounding / retrieval augmentation
- "The model takes in..." / "The pipeline works by..."
- "I use Claude API here because..." / "The prompt is structured as..."

### Practice Questions
1. Walk me through how you built an AI agent — what was the architecture?
2. What is RAG and when would you use it instead of fine-tuning?
3. How do you handle hallucinations in a production AI system?
4. What is your approach to prompt engineering for consistent output?
5. How do you evaluate the quality of an AI pipeline?

## Topic 7 — Project Storytelling (STAR Method)
**Weeks 13–14 | Daily practice: 30 min**

### Why This Topic
Behavioral and project-based interviews are as common as technical ones at senior level. STAR = Situation, Task, Action, Result — the universal structure international interviewers expect.

### Content Outline
**STAR Structure:**
- **Situation:** Context — what was the problem or challenge?
- **Task:** Your responsibility — what were you expected to do?
- **Action:** What you actually did — specific steps, decisions, tools
- **Result:** The measurable outcome — what changed, improved, or was achieved?

**Projects to prepare as STAR stories:**
1. Claude API affiliate content agent
2. Pinterest SaaS automation tool
3. Cloud Solution Architect AI agent (AWS/GCP)
4. A difficult technical problem you solved at work
5. A time you made an important technical decision with trade-offs

### Key Vocabulary & Phrases
- "The challenge we faced was..."
- "My responsibility was to..."
- "I decided to approach it by..."
- "The outcome was... / As a result..."
- "One thing I learned from this was..."
- "If I were to do it again, I would..."
- "The main trade-off I considered was..."

### Practice Questions
1. Tell me about a challenging project you led from start to finish.
2. Describe a time you had to make a technical decision with incomplete information.
3. Tell me about a time you built something under time pressure.
4. Describe a project where you had to learn a new technology quickly.
5. Tell me about a time your approach didn't work and what you did.

## Topic 9 — Team Communication & Code Review
**Weeks 17–18 | Daily practice: 30 min**

### Content Outline
**Daily remote work communication scenarios:**

1. **Daily standup** — what you did, what you'll do, blockers
2. **Code review** — giving and receiving feedback professionally
3. **Technical proposals** — presenting a solution to the team for discussion
4. **Async written communication** — Slack, GitHub comments, documentation
5. **Asking for help** — framing questions clearly and efficiently
6. **Status updates** — communicating progress and blockers to stakeholders

### Key Phrases by Scenario

**Standup:**
> "Yesterday I worked on... Today I'm planning to... I'm currently blocked on... and I might need help from..."

**Code review (giving feedback):**
> "Have you considered...?" / "This looks good — one suggestion would be..." / "I'm not sure I understand the reasoning here, could you explain?" / "This might cause issues when..."

**Code review (receiving feedback):**
> "Good point, I'll update that." / "My reasoning was X, but I see your concern — let me reconsider." / "Could you elaborate on what you mean by...?"

**Proposing a solution:**
> "I'd like to propose an approach for..." / "The problem we're solving is..." / "I've considered two options — here are the trade-offs..." / "I'm open to other ideas."

**Asking for help:**
> "I've been working on X for a few hours. I've tried Y and Z. The behavior I'm seeing is... Any ideas?"

### Key Vocabulary
- Blocker / dependency / blocked on / waiting for
- Pull request / PR / diff / merge / review
- LGTM (looks good to me) / nit / suggestion
- Async / sync / follow up / circle back
- Stakeholder / alignment / sign-off / sign-off

## Topic 10 — Salary Negotiation & Offer Discussion
**Weeks 19–20 | Daily practice: 30 min**

### Why This Topic Is Critical
Many engineers lose $500–$1,000/month by not negotiating. At $3,000/month current, the gap to $5,000 is not just technical — it is also this conversation.

### Content Outline
1. **Stating your expectation clearly** — when asked, how to answer
2. **Responding to an offer** — not accepting or rejecting immediately
3. **Negotiating upward** — making the case, using competing offers or market data
4. **Discussing total compensation** — base, equity, benefits, remote allowance
5. **Closing professionally** — how to accept, decline, or ask for time

### Key Phrases

**When asked about salary expectations:**
> "Based on my experience and the scope of this role, I'm targeting $X–$Y per month. I'm open to discussing the full package."

**When receiving an offer:**
> "Thank you for the offer — I'm genuinely interested in this role. Could I have a couple of days to review the details?"

**When negotiating:**
> "I'm very excited about this opportunity. The offer is close to what I had in mind, but based on my experience with LLM integration and AI pipeline design, I was hoping we could get to $X. Is there flexibility there?"

**If they say no room to negotiate:**
> "I understand. Are there other aspects of the package — like a remote equipment allowance or an earlier performance review — that might be adjustable?"

### Key Vocabulary
- Base salary / total compensation / equity / benefits
- Counter-offer / expectation / flexibility / room to move
- "I'm very excited about this role..."
- "Based on market research..."
- "Is there flexibility on..."
- "Could we revisit this at the 6-month mark?"

## Notes for Review

- If a topic feels underdeveloped after 2 weeks, extend by 3–5 days before moving on. Fluency over schedule.
- The warm-up (Topic 1 self-introduction) should be spoken every single day for all 180 days — this becomes the foundation of everything.
- Topic 11 (Kids English) is not optional practice — it is your daily immersion environment. Use it consistently.
- When you start getting interviews, shift 50% of daily time to mock interview practice using the practice questions from all topics.
